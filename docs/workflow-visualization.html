Phase 4 — Orchestrator

Opus (orchestration)

Responsibility

Phase 6 — Validation
Phase 6 — Quality Gate
QA Validator
Haiku (fast systematic checks)

Validation checklist

  • Every acceptance criterion has a passing test
  • API responses match design document exactly
  • Data model invariants enforced (state machines)
  • Components: loading / error / empty states present
  • No skipped tests, no .only filters
  • Coverage meets thresholds
  • No regressions in previously passing tests

On failure — rework loop

  • Design flaw → escalate to Architect
  • Implementation bug → back to Developer
  • After 3 cycles → escalate to Architect

Rules

  • testing-enforcement
  • testing-standards

Phase 7 — Security (conditional)
Phase 7 — Security Gate
Security Reviewer
Sonnet

When invoked (mandatory)

  • Auth or session management changes
  • Authorisation / permission gate changes
  • User input handling (forms, uploads, params)
  • Database queries with user-controlled data
  • Payment / financial logic
  • New third-party dependencies
  • Webhook receivers

OWASP coverage

  • Input validation (allowlist, type, length)
  • Output encoding (XSS, SQL injection)
  • Auth (bcrypt, session tokens, rate limiting)
  • Authz (default deny, IDOR prevention)
  • Secrets (never in source, logs, or responses)
  • Dependencies (CVE audit)

Blocks release on

  • Critical or High severity findings
No Critical or High security findings before deploy
Phase 8 — Deployment
Phase 8 — Deploy
SRE / DevOps
Sonnet

Prerequisites (all required)

  • QA validation: PASSED in handoff log
  • Security: approved (or not triggered)
  • docs/deployment/deployment-config.md filled in

Deployment phases

  • Phase 1: Architecture confirmed
  • Phase 2: App prepared (env vars, health check, logs)
  • Phase 3: Containerise (Dockerfile, verify, push)
  • Phase 4: Provision infrastructure (IaC plan → confirm → apply)
  • Phase 5: Secrets to chosen store
  • Phase 6: CI/CD pipeline
  • Phase 7: Smoke test + documentation

Invariants

  • Never choose cloud provider — uses deployment-config.md
  • IaC: plan first, wait for confirmation, then apply
  • Health check fails → rollback immediately
  • No secrets in images, source, or pipeline definitions

  • container-deployment-security

🛠 Setup Required — Fill In Before Using

CLAUDE.md  REQUIRED
Project name, tech stack, architecture decisions, common commands. Fill all placeholders before the first session.
docs/testing/testing-strategy.md  REQUIRED
Choose testing framework and define test commands. Required before first implementation session. Copy from testing-strategy-template.md.
.claude/hooks/stop-run-tests.sh  REQUIRED
Uncomment or set your full test suite command. Runs automatically after every Claude response.
.claude/hooks/post-edit-run-fast-tests.sh  REQUIRED
Uncomment your fast unit test command. Runs automatically after every file edit.
docs/design/ui-design.md  FRONTEND
UI design choices, color tokens, component library decisions. Required before any frontend session. Template at docs/design/ui-design-template.md.
docs/deployment/deployment-config.md  OPTIONAL
Cloud provider, environments, IaC tool, CI/CD choices. Only needed when you are ready to deploy. Copy from deployment-template.md.

📡 Inter-agent Communication

docs/memory/project-memory.md
Read at the start of every session by every agent. Stores: special requirements (REQ-NNN), session logs, known gotchas, pending work.
docs/memory/agent-handoff.md
Written when an agent finishes a task. Contains: what was built, API shapes, files changed, what the next agent needs to know. Enables parallel agents to stay in sync.
docs/memory/agent-calls.log
Auto-generated by hook. Timestamped record of every Agent and Skill invocation — used for audit and debugging.
docs/design/
data-model-*.md, architecture-*.md, api-*.md, ui-design.md — the contracts all agents implement against.
